## RESEARCH DOSSIER

### 1. Aider — Repo Map (tree-sitter + PageRank, zero LLM cost)
- Code: https://github.com/Aider-AI/aider
- Docs: https://aider.chat/docs/repomap.html · https://aider.chat/2023/10/22/repomap.html
- Standalone reimplementation (MCP server): https://github.com/pdavis68/RepoMapper
- Mechanism: tree-sitter parses every source file into an AST, extracting definitions (functions,
  classes) and references (calls, imports). Builds a directed graph — nodes are files/symbols, edges
  are cross-file references. Runs PageRank over this graph to score importance, using a *personalized*
  PageRank (50x multiplier for symbols referenced by files currently open/being edited) so ranking
  adapts to what the user is actively working on. Selects top-ranked symbols to fit a token budget via
  binary search. Caches tree-sitter parse results in a SQLite disk cache keyed by `(path, mtime)`.
- Zero LLM calls required to build the map itself — pure static analysis.

### 2. RepoGraph — Enhancing AI Software Engineering with Repository-level Code Graph
- Paper: arXiv 2410.14684 · Code: https://github.com/ozyyshr/RepoGraph
- Represents the repo as a line-level graph capturing structural/relational info across files, used to
  augment LLM context for repo-level completion and issue resolution instead of flat chunk retrieval.
  Performs subgraph retrieval — extracting an "ego-network" of relevant lines and their neighbors —
  rather than summarizing the whole repo up front. Evaluated as a plug-in module across multiple
  SWE-bench methods.

### 3. CodexGraph (Liu et al., 2024)
- Paper: arXiv 2408.03910 · Application demo: https://github.com/modelscope/modelscope-agent/tree/master/apps/codexgraph_agent
- Indexes the repo into a Neo4j graph database with a unified schema (MODULE, CLASS, FUNCTION, METHOD,
  FIELD, GLOBAL_VARIABLE nodes; CONTAINS, HAS_METHOD, INHERITS, USES, CALLS edges). An LLM agent
  translates natural-language questions into Cypher queries against this graph, executes them, and
  synthesizes results — enabling precise multi-hop lookups ("find all classes inheriting from X and
  overriding method Y").

### 4. KGCompass — repository-aware knowledge graph for automated program repair
- Paper: arXiv 2503.21710 · Code: https://github.com/GLEAM-Lab/KGCompass
- Links repo artifacts (issues, PRs) to codebase entities (files, classes, functions) via "entity path
  tracing," narrowing a bug's search space down to roughly the 20 most relevant functions. State-of-the-art
  single-LLM repair accuracy on SWE-bench Lite at ~$0.20 per repair. Runs on Neo4j via Docker Compose.

### 5. CGM — Code Graph Model (NeurIPS 2025)
- Paper/OpenReview: id `b98ODdeYq5` — no public code repo confirmed at time of writing; check the
  OpenReview page directly for any release.
- Goes beyond feeding graph info as text: integrates repo code-graph structure directly into the LLM's
  attention mechanism via a specialized adapter mapping graph node attributes into the LLM's input
  space, paired with an "agentless" graph-RAG framework. Represents the current technical ceiling of
  "graph-aware LLM" research — likely out of scope to reimplement, useful to know it exists.

### 6. HCAG — Hierarchical Abstraction and Retrieval-Augmented Generation on Theoretical Repositories with LLMs
- Paper: arXiv 2603.20299 — no public code repo confirmed at time of writing.
- **The closest thing found to "PageIndex, but for code repos."** Phase I (offline): recursive,
  bottom-up LLM summarization. Leaf files get LLM-generated summaries (key functionality, core logic,
  inputs/outputs, dependencies). Directory (non-leaf) nodes aggregate children's summaries into a
  higher-level summary describing the module's purpose/architecture. Recurses to the repo root. Has a
  configurable **compression-depth parameter (C)**: directories deeper than C get a placeholder
  ("to be detailed") with a pointer to underlying content, expanded on-demand only when later accessed
  during retrieval. Directly solves the "summarizing everything is too expensive at scale" problem.

### 7. Code-Craft / HCGS — Hierarchical Graph-Based Code Summarization for Enhanced Context Retrieval
- Paper: arXiv 2504.08975 — no public code repo confirmed at time of writing.
- Combines structural code graphs with LLM summarization at varying granularity, explicitly addressing
  the weakness that pure graph approaches "lack the capability to abstract details at varying levels of
  granularity." Related-work section is a useful map of the field: cites RepoUnderstander, GraphCoder
  (statement-level Code Context Graph combining control-flow + dependency relationships), CodexGraph,
  and RepoHyper as comparable multi-level-graph systems.

### 8. RAPTOR — Recursive Abstractive Processing for Tree-Organized Retrieval (ICLR 2024)
- Code (official): https://github.com/parthsarthi03/raptor
- General-text RAG, not code-specific — the origin of "recursively cluster and summarize chunks
  bottom-up into a tree" that HCAG and others adapt for code. Builds a multi-layer summary tree from
  leaf chunks upward; retrieval can pull from any layer (leaf = detail, top = global overview).

### 9. GraphRAG (Microsoft) and the LightRAG/community-summary family
- Microsoft GraphRAG — Paper: arXiv 2404.16130 ("From Local to Global...") · Code: https://github.com/microsoft/graphrag
- LightRAG (lighter/cheaper alternative, skips precomputed community summaries) — Code: https://github.com/HKUDS/LightRAG
- Curated list of the wider GraphRAG ecosystem (Fast GraphRAG, nano-graphrag, ApeRAG, Graphiti, DIGIMON,
  HuixiangDou2, GraphRAG-SDK, and more): https://github.com/DEEP-PolyU/Awesome-GraphRAG
- Mechanism (Microsoft GraphRAG): extracts an entity-relationship graph via LLM, runs hierarchical
  community detection (Leiden algorithm) to cluster the graph at multiple levels, generates LLM
  summaries per community bottom-up (leaf communities summarized first by prioritizing high-degree
  edges until a token budget is hit; higher-level communities summarize their sub-communities'
  summaries). At query time, uses a **map-reduce process**: community summaries are shuffled and
  chunked, partial answers generated per chunk, then reduced into a final answer — built for "global"
  questions across a corpus too large for one context window.

### 10. RepoUnderstander / Alibaba LingmaAgent
- Paper: arXiv 2406.01422 (LingmaAgent) — no public code repo confirmed (deployed internally in
  Alibaba's TONGYI Lingma IDE assistant, not released as standalone OSS as far as found).
- Top-down condensation of repo info into a knowledge graph, then **Monte Carlo Tree Search (MCTS)**
  guides an agent's exploration of that graph — the agent plans which parts of the repo to examine
  *before* acting, instead of having everything pre-summarized up front. An alternative philosophy to
  HCAG: build a lighter graph and let a search-guided agent decide where to look on demand.

### 11. LocAgent — Graph-Guided LLM Agents for Code Localization (ACL 2025)
- Paper: arXiv 2503.09089 · Code: https://github.com/gersteinlab/LocAgent
- Parses codebases into directed heterogeneous graphs (file/class/function nodes; import/invoke/inherit
  edges), giving LLM agents a lightweight structure to do multi-hop search/localization over. Includes a
  direct comparison table against CodexGraph, RepoUnderstander, and OrcaLoca on relation/node-type
  coverage and traversal strategy. Reports an ~86% cost reduction vs. proprietary-model baselines while
  reaching ~92.7% file-level localization accuracy with a fine-tuned open model.

### 12. RANGER — Repository-Level Agent for Graph-Enhanced reasoning
- Paper: arXiv 2509.25257 — no public code repo confirmed at time of writing.
- Two-path design: simple queries resolved directly; complex ones trigger MCTS-based graph exploration.
  Builds file-level graphs first via tree-sitter (explicitly avoiding Python-specific tools like Jedi/
  Parso — genuinely language-agnostic), then stitches them into a unified repo-level graph: intra-file
  structure resolved before cross-file dependencies. Good model for a multi-language, non-Python repo.

### 13. Understand-Anything (open-source, actively maintained, ~14.7k GitHub stars as of mid-2026)
- Code: https://github.com/Lum1104/Understand-Anything
- The most directly comparable *working, polished* system to what we're trying to build. A Claude Code
  / multi-platform plugin running a multi-agent pipeline: `project-scanner` (discover files/languages/
  frameworks) → `file-analyzer` (extract functions/classes/imports, produce graph nodes/edges,
  parallelized up to 5 concurrent, 20-30 files/batch) → `architecture-analyzer` (identify layers: API/
  Service/Data/UI/Utility) → `tour-builder` (dependency-ordered guided walkthroughs) →
  `graph-reviewer` (validate graph completeness/referential integrity) → optional `domain-analyzer`
  (extract business domains/flows/process steps) → optional `article-analyzer` (for wiki-style KBs).
  Output is a single JSON knowledge graph meant to be **committed to git** so teammates skip re-running
  the pipeline. Supports incremental updates (only re-analyzes changed files via a post-commit hook),
  diff-impact analysis, and ships an interactive dashboard.

### 14. Code-Graph-RAG (open-source, ~2.3k+ stars)
- Code: https://github.com/vitali87/code-graph-rag
- Another directly comparable working system. Parses multi-language monorepos (12 languages) with
  tree-sitter into a unified graph schema, stores the graph in **Memgraph**, and exposes it as an
  **MCP server** so Claude Code (or any MCP client) can query and edit the codebase via natural
  language translated into Cypher. Also does semantic code search (intent-based, via UniXcoder
  embeddings) on top of the graph, and supports AST-targeted surgical code edits. Useful as a reference
  for the "graph DB + natural-language-to-Cypher" pattern if that route is chosen over precomputed
  summaries.

### 15. Knowledge Graph Based Repository-Level Code Generation (ICSE 2025)
- Paper: arXiv 2505.14394 — no public code repo confirmed at time of writing.
- Hybrid graph-based code retrieval pipeline: builds a KG from the repo, does hybrid (graph + semantic)
  retrieval to find a relevant subgraph, feeds that subgraph to the LLM for code generation, tracking
  inter-file modular dependencies to keep generated code consistent with existing conventions.

### 16. Awesome-Repo-Level-Code-Generation (curated bibliography, updated through April 2026)
- Code/list: https://github.com/YerbaPage/Awesome-Repo-Level-Code-Generation
- A continuously updated list of repo-level code generation / issue-resolution papers (SWE-agent,
  AutoCodeRover, Agentless, CodeR, AEGIS, and dozens more). Not all relevant to knowledge representation
  specifically — filter for "graph" / "knowledge" / "understand" in titles.

### 17. Adjacent, different goal — legacy code modernization with LLMs
- No code repos (these are research papers / industry blog posts, not tools): PromptLayer's review of
  "Leveraging LLMs for Legacy Code Modernization" (MUMPS/assembly documentation study); industry
  write-ups from ModLogix and createq.com.
- About *understanding old code well enough to safely change it* (generating "living documentation,"
  flagging obsolete dependencies/anti-patterns) — not about mining code for new product ideas. Different
  goal, but file/module summarization mechanics overlap with what's needed here.

### 18. Adjacent, different problem entirely — commercial "GenAI use case discovery" tooling
- No code repos (commercial SaaS, not open tools): Fluency (usefluency.com), AWS Marketplace's
  "Generative AI Use Case Discovery" service, IDC's GenAI Use Case Discovery Tool.
- All mine **business process / operational data** (tickets, meetings, workflows) to rank automation
  opportunities by ROI. None analyze source code directly.
- **Notable negative finding:** no existing tool was found that statically analyzes source code itself
  (as opposed to business process logs) to surface customer-facing AI/ML opportunity hooks. This
  specific angle — code-level static analysis + LLM reasoning, categorized into product-opportunity
  buckets — appears genuinely under-explored. Prior art splits into "code-graph-for-coding-tasks"
  (items 1-16 above) or "business-process-mining-for-AI-strategy" (this item), not the intersection.

### 19. PageIndex (the original inspiration for this whole investigation)
- Code: https://github.com/VectifyAI/PageIndex
- Vectorless, reasoning-based RAG for long PDFs. Builds a hierarchical "table of contents" tree (each
  node: `title`, `node_id`, `start_index`/`end_index` page pointers, LLM-generated `summary`, child
  `nodes`), then does reasoning-based tree search at query time instead of vector similarity search. No
  chunking, no vector DB; retrieval traces back to an exact page/section. This is the explicit pattern
  we want adapted to code — and HCAG (#6) is essentially this same idea independently re-derived for
  code repositories instead of PDFs.

This is provided as context only — **you do not need to build on top of it.** If a fresh,
ground-up design serves the goal better than extending the existing scanner, propose that instead.
What matters is the underlying goal: turn a large repository into a persistent, reusable knowledge
representation, then mine that representation for AI/ML product opportunities, better and cheaper than
re-reading raw chunked source every time.

## YOUR TASK

Given everything above, and the actual repos/papers available to you in folder context (read them
directly, don't just rely on these summaries):

1. **Propose a design** for turning a large multi-module repository into a persistent, reusable
   knowledge representation — built once, queried/mined repeatedly. Don't default to copying any one
   approach above wholesale; reason about which mechanisms from which systems actually solve our
   problem, and which are solving a different problem (bug localization, code generation, repair) that
   doesn't transfer cleanly.

2. **Explicitly weigh the trade-offs** between the different schools of thought found: deterministic
   structural graphs with no LLM cost (Aider), hierarchical bottom-up LLM summarization (HCAG, GraphRAG,
   RAPTOR), knowledge-graph-with-agentic-querying (CodexGraph, Code-Graph-RAG, LocAgent), and
   MCTS-guided on-demand exploration (RepoUnderstander/LingmaAgent, RANGER). State which you'd use, in
   what combination if any, and why — including where you'd diverge from what these papers/tools did
   because our task (opportunity mining) differs from theirs (code gen, repair, localization, chat-based
   codebase Q&A).

3. **Address persistence and cost concretely**: how should the knowledge representation be stored
   (format/structure, not just "a database"), how does it stay cheap to rebuild after small code
   changes, and how does a later opportunity-mining pass avoid re-reading the entire repo every time.

4. **Address how opportunity-mining itself should work** on top of whatever representation you propose
   — what does the actual mining prompt/process look like, and does it need different modes (e.g.
   "scan everything" vs. "dig into this one area on request")?

5. **Decide and justify** whether this should extend the existing AI Opportunity Scanner or be built as
   a new project from scratch, and why.

6. **Call out open questions or risks** you're not confident about, and what you'd want to test on a
   real repo before committing to the design.

Do not organize your answer as a phased rollout plan or implementation roadmap. This is a design and
reasoning exercise — state your recommended approach and the reasoning behind it, and create whatever
plan you think is actually right once you've reasoned through it.
